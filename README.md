# Liquid Glass Calendar

Liquid Glass Calendar is a Vite + React calendar app that can ship in two ways:

- as a static web app deployed from `dist/`
- as a Capacitor-wrapped iPhone app from the same web build

The current codebase is web-deployable now, with iPhone PWA support prepared and the native iOS wrapper still available.

## Design direction

- The UI is being steered toward a Toss-like direction: crisp white cards, strong spacing, restrained blue emphasis, and simpler hierarchy over heavy glass effects.
- Existing component names may still contain older "glass" terminology, but the current visual system favors flatter premium surfaces.

## Current deployment posture

- Build target: static files generated by `npm run build` into `dist/`
- Recommended free host: Vercel
- PWA shell: manifest, Apple web app meta tags, and app icons are present
- Data mode: local-first by default, with optional Supabase sync
- Limitation: no service worker is configured, so this is installable/standalone but not offline-first

## Quick start

```bash
npm install
cp .env.example .env.local
npm run build
npm run test -- --run
npm run preview
```

For the simplest production deploy, keep the default local-first env values from `.env.example`.

## Environment variables

The app can deploy with no backend credentials when using local mode.

- `VITE_APP_NAME`: app title shown in the UI
- `VITE_SYNC_PROVIDER`: `local` or `supabase`
- `VITE_SHARED_CALENDAR_NAME`: display name for the calendar
- `VITE_SHARED_CALENDAR_SLUG`: stable key used for shared calendar scope
- `VITE_SUPABASE_URL` / `VITE_SUPABASE_ANON_KEY`: only needed when `VITE_SYNC_PROVIDER=supabase`
- `VITE_ENABLE_GOOGLE_CALENDAR` / `VITE_GOOGLE_CLIENT_ID`: optional Google Calendar integration

Important: auth in this repo is still local-browser auth. Supabase data sync is scaffolded, but Supabase Auth is not wired in yet.
Read [SECURITY.md](/Users/dhwoo/Documents/web_calendar/SECURITY.md) before treating this as a production-ready auth or shared-data setup.

## Vercel deployment

This repo already contains a `.vercel/` link locally and now includes a minimal `vercel.json` so the static build assumptions are explicit.

### Recommended path

1. Import the repository into Vercel.
2. Let Vercel detect the project as Vite.
3. If Vercel asks for overrides, use:
   - Build Command: `npm run build`
   - Output Directory: `dist`
4. Add only the environment variables you actually use in production.
5. Deploy and verify the checklist in `DEPLOY_CHECKLIST.md`.

## Security posture summary

- Local auth is convenience-only and stored in browser `localStorage`.
- Google Calendar tokens are intended to remain in memory during the session.
- Supabase shared-sync mode is useful for MVP collaboration, but without Supabase Auth it should not be treated as a secure multi-user boundary.
- Operational notes for CLI tooling, including OpenClaw notification usage, are documented in `SECURITY.md`.

### Notes for optional integrations

- Google Calendar: add your production origin to Google OAuth authorized JavaScript origins before enabling `VITE_ENABLE_GOOGLE_CALENDAR=true`.
- Supabase sync: set your production site URL in Supabase and provide `VITE_SUPABASE_URL` and `VITE_SUPABASE_ANON_KEY`.

## iPhone PWA readiness

The web app is prepared for home-screen installation on iPhone:

- `public/manifest.json` defines standalone display and icons
- `index.html` includes Apple mobile web app meta tags
- layout code already handles safe areas and mobile viewport height

Current limitation:

- there is no service worker, so users should expect a standalone shell, not reliable offline caching

Deployment assumption:

- the current asset paths and manifest `start_url` assume the app is served from the site root `/`
- if you later deploy under a subpath, update the Vite `base` setting and the manifest URLs together

## Other free hosting options

If you do not want Vercel, the app can also ship as a plain static build on Netlify, Cloudflare Pages, or GitHub Pages. The important requirement is serving the generated `dist/` directory over HTTPS.

## Native iOS wrapper

The Capacitor workflow is still available:

```bash
npm run cap:sync
npm run cap:open:ios
```

Or:

```bash
npm run ios
```

Use `IOS_RELEASE_CHECKLIST.md` for the native App Store/TestFlight handoff and `DEPLOY_CHECKLIST.md` for the web/PWA release path.
